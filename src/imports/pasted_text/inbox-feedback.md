How the inbox works today (visual audit)
The layout is a three-zone structure: a global left nav (Today, Ask, Tasks, Inbox, etc.), a secondary column with two groupings — a folder/status set (Needs attention 37, Sent 8, Flagged 7, Reviewed 11, All mail 54, Archived 1) and a PATHWAYS set (Needs reply, Quotes, Quality/PCN, Logistics, FYI, Noise, each with counts) — and then the main list. The main list groups threads under date headers (Today / Yesterday), and each row shows an avatar, sender addresses, a message-count badge, subject, an AI one-line summary (orange sparkle icon), and a row of tags: a category chip (Quote / Logistics / FYI / Noise), a "Needs review" chip, an "Attachment" chip, and sometimes "High" or "Replied." High-priority rows get a red left border accent. Top right has "New email"; the header reads "MERIT OEM · LIVE FIREHOSE."
Opening a thread replaces the whole main pane with a detail view: a back link, subject, message count, Forward / Flag / Archive actions, an AI Summary card, a Suggested Attachments card, a TRIAGE bar with category buttons plus "Mark reviewed," then a vertical stack of message cards, and a reply composer at the bottom with a "Draft with AI" field, rich-text toolbar, Reply-all toggle, and "Send to all."
Confirmed issues and recommendations
1. Triage only exists inside the thread, not from the list. I hovered over list rows and no inline actions appear — the category and "Mark reviewed" controls live only in the thread detail. This matches what you described. The category chips on list rows also appear to be display-only labels, not interactive. Recommendation for design: add hover/selected-row quick actions on each list row (assign category, mark reviewed, archive, flag, snooze), ideally with keyboard shortcuts and multi-select for bulk triage, so a user can clear the "Needs attention 37" queue without opening every thread.
2. Opening a thread doesn't reliably clear the unread/attention state. The threads carry a blue unread dot and a "Needs review" chip, but these are separate from actually reading the thread — the only thing that changes state is the explicit "Mark reviewed" button inside the thread. So opening ≠ triaged, which is the friction you flagged. Design should decide and make explicit the distinction between three states — unread (auto-clears on open), needs review/triage (an AI-flagged action state), and reviewed/done — and show them as visually distinct treatments rather than overloading one dot. Auto-clearing "unread" on open while keeping "needs review" as a deliberate action would resolve most of the confusion.
3. Adding new triage categories. The categories (Needs reply, Quotes, Quality/PCN, Logistics, FYI, Noise) appear fixed in both the pathways sidebar and the in-thread triage bar. There's no "+" or management affordance. Design should add an "Add category / Manage pathways" entry (likely surfaced in the pathways list and/or Settings), and the in-thread triage bar should stay in sync with whatever categories are defined so the two lists never diverge.
4. Thread mapping and internal-vs-external separation — this is the biggest one. In the ReCurve thread, external customer emails (kenny@recurvemedical.com) and internal Merit-only discussion (rjenkins saying "I'll let Chad respond… he's more familiar with it," and Jordan's internal costing notes to rbest@merit.com) are all rendered as identical stacked cards in a single flat timeline. The only differentiator is a tiny "Received"/"Sent" text badge — there's no visual grouping, color, or lane that says "this sub-conversation is an internal side-thread that branched off the customer thread but still belongs to it." For a triage tool this is exactly where clarity is needed. Recommendations for design: distinguish external vs. internal messages with clear treatments (e.g., a lane/indent, a subtle background tint, or an "Internal" label), and consider a "branch/side-conversation" grouping — collapsible internal segments shown as an offset chain or a side rail — so the customer-facing spine of the thread reads cleanly while internal deliberation is visibly attached-but-separate. A participants/thread-map summary at the top (who's external, who's internal, where the conversation forked) would reinforce this.
5. Formatting, spacing, and noise. The message bodies dump raw pasted signature/table data repeatedly ("Pasted table data:… Richard Best Director… UNDERSTAND. INNOVATE. DELIVER. TM" appears many times), and there are large empty vertical gaps inside cards where signatures were stripped. This makes each message very tall and the real content hard to find. Design should collapse quoted history and signatures behind a "…" expander (Gmail-style trimmed content), suppress or de-emphasize repeated boilerplate, and tighten card padding so the actual message and its AI summary lead. The message cards also lack per-message actions (reply to one message, view original) — worth adding.
6. Smaller notes for design. The category chip, "Needs review" chip, and "Attachment" chip all sit at the same visual weight in list rows, so priority isn't scannable — consider a hierarchy (priority > category > status > metadata). The "LIVE FIREHOSE" framing implies high volume, which strengthens the case for list-level bulk triage and saved/smart views. And the reply composer's "Send to all" with 8 recipients on a mixed internal/external thread is a place where the internal/external distinction matters for safety — design may want a recipient-review step there.
